  • JSON Web Token (JWT)执行的一种基于 JSON 的开放标准(RFC 7519)，它定义了一种紧凑的、自包含的方式，用于作为 JSON 对象在各方之间安全地传输信息。该信息可以被验证和信任，因为它是数字签名的。

# 基于服务器的身份认证(session)

采用 cookie 机制，会有一台服务器专门做池子，所有的服务器都还要往这个服务器去询问，这个 cookie 是不是有效的 如果服务器多，那个 cookie 服务器就会很累 jwt 就不需要主服务器了，每个服务器都可以独立解密

  • 原理

  • http 协议本身是一种无状态的协议，用户提供了用户名和密码来进行用户认证，下一次请求时，用户还要再一次进行用户认证才行。

  • 为了能识别是哪个用户发出的请求，在服务器存储用户登录的信息，在响应时传递给浏览器，告诉其保存为 cookie,下次请求时发送给服务器，这样服务器就能识别请求来自哪个用户了,这就是传统的基于 session 认证。

  • 采用 cookie 机制，会有一台服务器专门做池子，所有的服务器都还要往这个服务器去询问当前 cookie 的有效性，如果服务器多，那个 cookie 服务器就会很累。

  • 缺点

  • 随着不同客户端用户的增加，服务端的开销会明显增大，独立的服务器已无法承载更多的用户。

  • 扩展性: 用户认证之后，服务端做认证记录并保存在其内存中，用户下次请求还必须要请求在这台服务器上,才能拿到授权的资源，而在分布式的应用上则限制了负载均衡器的能力。这也意味着限制了应用的扩展能力。

  • CSRF 跨站请求伪造：因为是基于 cookie 来进行用户识别的, cookie 如果被截获，用户就会很容易受到跨站请求伪造的攻击。

# 基于 Token 的身份认证(JSON Web Token)

  • 特点
  • 基于 Token 的身份认证是无状态的，服务器或者 Session 中不会存储任何用户信息，服务器没有那个登记身份“池子”。
  • 优点
  • 基于 token 的鉴权机制不需要在服务端去保留用户的认证信息或者会话信息。用户不需要去考虑哪一台设备登录，这为扩展提供了便利。特别适用于分布式站点的单点登录（SSO）场景。
  • json 的通用性，所以 JWT 是可以进行跨语言支持的，像 JAVA,JavaScript,NodeJS,PHP 等很多语言都可以使用。
  • 因为有 payload 部分，JWT 可以在自身存储一些其他业务逻辑所必要的非敏感信息。
  • 便于传输，jwt 的构成非常简单，字节占用很小，非常便于传输的。
  • 缺点
  • jwt 一旦签发出去，无法回收回来，而 cookie 机制只要池子里把你抹杀，你的 cookie 直接作废
  • jwt 无法拉黑，因为这段字符串是固定的，一定有效
  • 这里的关键是，服务器想拉黑某个用户，拉黑不了，cookie 直接删了，登录就失效了，一般会混入时间戳来做时间有效期验证，但是如果时间尚未失效，就无法拉黑。
  • 没有混入 ip 地址或者设备信息时，客户可以手动把 jwt 复制走，去哪里都是登录状态
  • 安全性
  • 不应该在 jwt 的 payload 部分存放敏感信息，因为该部分是客户端可解密的部分。
  • 保护好 secret 私钥，该私钥非常重要。
  • 如果可以，请使用 https 协议

# JSON Web Token 应用场景

  • Authorization (授权) : 这是使用 JWT 的最常见场景。一旦用户登录，后续每个请求都将包含 JWT，允许用户访问该令牌允许的路由、服务和资源。单点登录是现在广泛使用的 JWT 的一个特性，因为它的开销很小，并且可以轻松地跨域使用。
  • Information Exchange (信息交换) : 对于安全的在各方之间传输信息而言，JSON Web Tokens 无疑是一种很好的方式。因为 JWT 可以被签名，例如，用公钥/私钥对，你可以确定发送人就是它们所说的那个人。另外，由于签名是使用头和有效负载计算的，您还可以验证内容没有被篡改。

# JSON Web Token 由三部分组成，用圆点(.)连接。

  • 三部分是三段字符串。用小数点分割
  1. Header
  2. Payload
  3. Signature
  • 一个典型的 JWT 看起来是这个样子的：
  • xxxxx.yyyyy.zzzzz
  • 第一段和第二段经过加密会获得第三段
  • 解密出来就是你的身份信息了
  • 服务器只要知道加密解密方式就可以了
  • 加密里会混入时间戳来保证整个字符串独一无二，并且也能通过时间来保证 jwt 的时间有效性，时间到期之前，服务器无法做出拉黑行为
  • 客户手动把 jwt 复制走，去哪里都是登录状态，所以 jwt 一般还要混入 ip 地址，用 ip+时间戳+用户 id 来加密

  • 有些应用是有和浏览器合作的，让 chrome 动用系统权限帮忙读取你电脑的文件
  • 例如：
  • git 官网让 chrome 去读你电脑的 git config

# 流程

  • 用户使用用户名密码来请求服务器
  • 服务器进行验证用户的信息
  • 服务器通过验证发送给用户一个 token
  • 客户端存储 token，并在每次请求时附送上这个 token 值
  • 服务端验证 token 值，并返回数据
  • 这个 token 必须要在每次请求时传递给服务端，它应该保存在请求头里， 另外，服务端要支持 CORS(跨来源资源共享 Access-Control-Allow-Origin)策略。

# 实现

httpRequest = (url, type = "get", data = {}, errorTitle = "") => {
  const method = type.toLowerCase();
  // 如果token存在就携带在请求头一并发送出去
  if (localStorage.token) {
    this._axiosInstance.defaults.headers.common["Authorization"] =
      localStorage.token;
  }
  return new Promise((resolve, reject) => {
    instance
      .request({
        url,
        method,
        params: method === "get" ? data : {},
        data: method !== "get" ? data : {},
      })
      .then((res) => {
        console.log(res.data);
        //这里的status是和后端约定的
        if (res.data.status === "ok") {
          if (res.data.token) {
            localStorage.token = res.data.token;
          }
        }
        resolve(res.data);
      })
      .catch((err) => {
        // 出错时打印错误
        const tip = "请求" + errorTitle + "失败";
        console.log(tip);
        reject(err);
      })
      .finally();
  });
};

# 注意事项

  • 登录验证：请求前要设置请求头带上 token
  • 登录成功：响应成功接到响应后要保存 token
  • 注销登录：注销成功响应后要将 token 手动删除
  • 由于登陆验证时设置了带上 token 的请求头，为了保证当不是登陆验证时不会把 token 请求头携带出去,每次非登陆验证请求前都要先把 token 请求头移除。
